Repository

Repository (repo)

A repository is the project folder that Git tracks. It stores the full version history, branches, and metadata for a project.

Considerations:
  • Keep a repo focused on one project or service.
  • Use .gitignore to exclude generated files and secrets.
  • Choose a clear README and branch strategy from the start.
Resources:
  • Git Basics: Getting a Git Repository
  • GitHub: Create a repository
Clone

Clone

Cloning downloads a repository copy to your machine, including commit history and branches. It creates a local copy linked to the remote.

Considerations:
  • Clone only when you need local access.
  • Use shallow clones for very large repositories.
  • Update remote URLs after renaming or moving repos.
Resources:
  • Git Clone docs
  • GitHub: Cloning a repository
Branch

Branch

A branch is a safe parallel timeline. It lets you develop features or fixes independently from the main code line.

Considerations:
  • Name branches clearly by feature, bug, or release.
  • Keep branch lifetimes short for easier collaboration.
  • Sync your branch frequently with main.
Resources:
  • Git Branching
  • GitHub branches guide
Commit

Commit

A commit is a recorded snapshot and message describing a change. It becomes part of the repo's history.

Considerations:
  • Write meaningful commit messages.
  • Commit small, logical units of work.
  • Stage only related files before committing.
Resources:
  • Git commits
  • How to write a Git commit message
Push / Pull

Push / Pull

Push uploads local commits to the remote. Pull downloads remote updates and merges them into your local branch.

Considerations:
  • Pull before you push to reduce conflicts.
  • Push only finished or review-ready work.
  • Use fetch when you want more control over remote changes.
Resources:
  • Git Push docs
  • Git Pull docs
Diff

Diff

A diff shows file changes between commits, branches, or working trees. It highlights additions, removals, and edits.

Considerations:
  • Inspect diffs before merging or reviewing code.
  • Keep changes focused so diffs stay readable.
  • Use diff tools to compare file versions visually.
Resources:
  • Git Diff docs
  • GitHub diff viewer
Merge

Merge

Merge combines one branch into another. It joins multiple timelines into a single history path.

Considerations:
  • Merge regularly to reduce conflict size.
  • Choose merge strategies intentionally.
  • Review branch changes before merging.
Resources:
  • Git Merge docs
  • GitHub merge methods
Rebase

Rebase

Rebase reapplies commits on top of a new base branch. It rewrites history so your work appears built from the latest main branch.

Considerations:
  • Avoid rebasing public history already shared with teammates.
  • Use rebase to keep history linear and tidy.
  • Resolve conflicts carefully while replaying commits.
Resources:
  • Git Rebase docs
  • Merging vs rebasing
Merge conflict

Merge conflict

A merge conflict happens when two branches change the same lines. Git pauses and asks you to resolve the conflict manually.

Considerations:
  • Resolve conflicts by understanding both sides.
  • Test after merging to ensure correctness.
  • Communicate with teammates when shared code is affected.
Resources:
  • Git merge conflict help
  • GitHub resolve merge conflicts
Pull Request

Pull Request (PR)

A pull request is a request to review and merge one branch into another. It's the main collaboration workflow on GitHub.

Considerations:
  • Add a clear description and link related issues.
  • Use draft PRs for work in progress.
  • Encourage reviewers with focused, small PRs.
Resources:
  • GitHub pull requests
  • How to make a pull request
Issue

Issue

An issue is a ticket for a bug, task, or feature request. It tracks work and discussion separate from code.

Considerations:
  • Write clear titles and use labels for priority.
  • Link issues to pull requests and commits.
  • Use templates to standardize reporting.
Resources:
  • GitHub issues
  • Issue tracking best practices
